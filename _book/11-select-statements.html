3  SELECT Statements

Now that your database environment is up and running, it is time to write some SQL. The most fundamental operation in SQL is the query: a question you ask the database, written in a structured way that the database engine can understand and answer. In this chapter, you will write your first queries against the countries database that ships with the course Docker image.

The countries table contains one row per country or territory, covering 215 entries from a 2022 snapshot of World Bank data. Each row records the country’s name, ISO codes, population, land area, world region, capital city, and geographic coordinates. It is a rich dataset for learning SQL because it mixes text columns, whole numbers, decimal numbers, and dates in a single table.

By the end of this chapter, you will know how to retrieve data from a table, choose which columns to display, compute new values from existing ones, filter rows by any condition you can describe, sort the results, and cap how many rows are returned.

3.1 The SELECT and FROM Clauses

Every query that retrieves data starts with a SELECT statement. The two required pieces are:

  • SELECT specifies which columns you want in the result.
  • FROM tells the database which table to look in.

3.1.1 Selecting All Columns

The quickest way to see everything a table contains is to use an asterisk (*) in place of column names. The asterisk is shorthand for “every column, in the order they are defined.”

SELECT *
FROM countries
LIMIT 5;
5 records
country iso2c iso3c yr population area lastupdated region capital longitude latitude
Afghanistan AF AFG 2022 40578842 652230 2024-12-16 South Asia Kabul 69.17610 34.5228
Albania AL ALB 2022 2777689 27400 2024-12-16 Europe & Central Asia Tirane 19.81720 41.3317
Algeria DZ DZA 2022 45477389 2381741 2024-12-16 Middle East & North Africa Algiers 3.05097 36.7397
American Samoa AS ASM 2022 48342 200 2024-12-16 East Asia & Pacific Pago Pago -170.69100 -14.2846
Andorra AD AND 2022 79705 470 2024-12-16 Europe & Central Asia Andorra la Vella 1.52180 42.5075

The LIMIT 5 at the end restricts the output to five rows. Without it, the query would return all 215 rows in the table. This is similar to the head() function in R or Python. You will learn more about LIMIT later in this chapter, but getting into the habit of adding it while exploring an unfamiliar table is a good practice.

3.1.2 Selecting Specific Columns

In most real situations, you will not want every column. Listing only the columns you need makes results easier to read and faster to retrieve from large tables.

SELECT country, capital, region, population
FROM countries;
Displaying records 1 - 10
country capital region population
Afghanistan Kabul South Asia 40578842
Albania Tirane Europe & Central Asia 2777689
Algeria Algiers Middle East & North Africa 45477389
American Samoa Pago Pago East Asia & Pacific 48342
Andorra Andorra la Vella Europe & Central Asia 79705
Angola Luanda Sub-Saharan Africa 35635029
Antigua and Barbuda Saint John’s Latin America & Caribbean 92840
Argentina Buenos Aires Latin America & Caribbean 45407904
Armenia Yerevan Europe & Central Asia 2969200
Aruba Oranjestad Latin America & Caribbean 107310

Column names are listed after SELECT, separated by commas. The database returns them in exactly the order you specify, regardless of how they are arranged inside the table.

Relying on SELECT * in production code is considered poor practice because it breaks silently when someone adds, removes, or reorders columns in the table. Naming your columns explicitly makes queries more predictable and easier to maintain.

3.2 Shaping Your Output

Beyond choosing which columns to include, SQL lets you rename columns, combine text from multiple columns, and compute new values on the fly. None of these operations modify the data in the table; they only affect what is shown in the result.

3.2.1 Column Aliases

By default, a column in your result inherits the name it has in the table. You can override this with the AS keyword, which assigns an alias to a column. Aliases are useful for giving computed values a meaningful name, shortening long column names, or improving readability.

SELECT
    country    AS name,
    iso3c      AS code,
    region     AS world_region,
    population AS population_2022
FROM countries;
Displaying records 1 - 10
name code world_region population_2022
Afghanistan AFG South Asia 40578842
Albania ALB Europe & Central Asia 2777689
Algeria DZA Middle East & North Africa 45477389
American Samoa ASM East Asia & Pacific 48342
Andorra AND Europe & Central Asia 79705
Angola AGO Sub-Saharan Africa 35635029
Antigua and Barbuda ATG Latin America & Caribbean 92840
Argentina ARG Latin America & Caribbean 45407904
Armenia ARM Europe & Central Asia 2969200
Aruba ABW Latin America & Caribbean 107310

The AS keyword is optional in PostgreSQL; you can write the alias directly after the column name without it. Using AS explicitly is recommended, however, because it makes the intent of the query immediately clear to anyone reading it.

3.2.2 Combining Text Columns

PostgreSQL uses the || operator to concatenate (join) strings together. This lets you build a single display value out of multiple columns, or add literal text between column values.

SELECT
    country || ' (' || iso3c || ')' AS display_name,
    capital,
    region
FROM countries;
Displaying records 1 - 10
display_name capital region
Afghanistan (AFG) Kabul South Asia
Albania (ALB) Tirane Europe & Central Asia
Algeria (DZA) Algiers Middle East & North Africa
American Samoa (ASM) Pago Pago East Asia & Pacific
Andorra (AND) Andorra la Vella Europe & Central Asia
Angola (AGO) Luanda Sub-Saharan Africa
Antigua and Barbuda (ATG) Saint John’s Latin America & Caribbean
Argentina (ARG) Buenos Aires Latin America & Caribbean
Armenia (ARM) Yerevan Europe & Central Asia
Aruba (ABW) Oranjestad Latin America & Caribbean

Here, three values are joined into one: the country name, a space and opening parenthesis, the three-letter ISO code, and a closing parenthesis. The result is a compact label like Afghanistan (AFG).

If any value being concatenated is NULL, the entire result of || becomes NULL. The countries table has a handful of rows where capital is missing; if you were to concatenate capital into a string without accounting for that, those rows would produce a NULL result. The COALESCE function, covered in a later chapter, handles this situation cleanly.

3.2.3 Arithmetic Operations

SQL treats numeric columns like variables in an algebraic expression. You can add, subtract, multiply, and divide columns and constants using standard arithmetic operators (+, -, *, /).

As an example, population density (people per square kilometer) is not stored in the table, but you can calculate it from population and area. Seven countries in the dataset have no recorded area, so the query filters them out first.

SELECT
    country,
    population,
    area,
    ROUND(population::numeric / area, 1) AS pop_per_sq_km
FROM countries
WHERE area IS NOT NULL
ORDER BY pop_per_sq_km DESC;
Displaying records 1 - 10
country population area pop_per_sq_km
Monaco 38931 2.084 18680.9
Singapore 5637022 718.000 7851.0
Bahrain 1524693 790.000 1930.0
Maldives 524106 300.000 1747.0
Malta 531511 320.000 1661.0
Bangladesh 169384897 130170.000 1301.3
Sint Maarten (Dutch part) 42139 34.000 1239.4
Bermuda 64749 54.000 1199.1
Barbados 282318 430.000 656.6
Mauritius 1262523 1997.000 632.2

ROUND(value, 1) rounds the result to one decimal place. The ::numeric cast converts the population column (stored as a whole-number bigint) to a decimal type before dividing, which is necessary to get a decimal result rather than integer division.

Monaco leads the list by a wide margin, followed by Singapore and Bahrain. The high density values for these small, densely settled territories illustrate why raw population numbers alone can be misleading without geographic context.

3.3 Exploring Data with DISTINCT and COUNT

Two features of the SELECT clause are especially useful when you first sit down with an unfamiliar database: DISTINCT, which removes duplicate values from your results, and COUNT, which tallies rows. Together they let you quickly understand what a table contains and how its data is distributed before you start filtering or transforming it.

3.3.1 DISTINCT

Adding the DISTINCT keyword immediately after SELECT tells the database to return only unique combinations of the specified columns, removing any duplicate rows from the result.

A common use is to see every unique value that appears in a categorical column. The countries table has 215 rows but only seven distinct world regions. Without DISTINCT, selecting the region column would return 215 rows with many repeated values. With it, you get the seven unique values once each:

SELECT DISTINCT region
FROM countries
ORDER BY region;
7 records
region
East Asia & Pacific
Europe & Central Asia
Latin America & Caribbean
Middle East & North Africa
North America
South Asia
Sub-Saharan Africa

DISTINCT applies to the full combination of columns listed, not just the first one. If you write SELECT DISTINCT region, capital, the result contains each unique pairing of those two values rather than unique regions alone.

SELECT DISTINCT is useful for a quick look at what values exist in a column, but it is not a substitute for proper data profiling. For large tables, counting how many rows correspond to each distinct value tells you much more than the list of values alone. GROUP BY, covered in a later chapter, is the right tool for that.

3.3.2 COUNT

COUNT is an aggregate function: rather than returning one row per input row, it collapses results into a single summary value. The most common form, COUNT(*), returns the total number of rows in the result.

SELECT COUNT(*) AS total_countries
FROM countries;
1 records
total_countries
215

Paired with a WHERE clause, COUNT(*) counts only the rows that satisfy the filter:

SELECT COUNT(*) AS south_asia_count
FROM countries
WHERE region = 'South Asia';
1 records
south_asia_count
8

COUNT(column_name), with a specific column instead of *, counts only rows where that column is not NULL. This makes it a convenient way to measure how complete your data is:

SELECT
    COUNT(*)       AS total_rows,
    COUNT(area)    AS rows_with_area,
    COUNT(capital) AS rows_with_capital
FROM countries;
1 records
total_rows rows_with_area rows_with_capital
215 208 209

The result shows that 208 of the 215 rows have a recorded area and 209 have a recorded capital, immediately revealing which columns have missing data and how many gaps exist.

COUNT(DISTINCT column) combines both ideas, counting how many unique values appear in a column:

SELECT COUNT(DISTINCT region) AS num_regions
FROM countries;
1 records
num_regions
7

COUNT belongs to a family of aggregate functions that summarize groups of rows into a single value. Others include SUM(), AVG(), MIN(), and MAX(). These become significantly more powerful when paired with GROUP BY, which computes a separate summary for each category in a column, such as the number of countries per region or the average population per continent. GROUP BY is covered in its own chapter.

3.4 Filtering Rows with WHERE

SELECT and FROM control which columns and tables you work with. The WHERE clause controls which rows appear in the result. Any row that does not satisfy the condition after WHERE is excluded from the output.

SELECT column1, column2
FROM table_name
WHERE condition;

3.4.1 Comparing Text Values

Use a single equals sign (=) to match a text value exactly. Text comparisons in PostgreSQL are case-sensitive by default, so 'South Asia' and 'south asia' are not the same.

SELECT country, capital, population
FROM countries
WHERE region = 'South Asia'
ORDER BY population DESC;
8 records
country capital population
India New Delhi 1425423212
Pakistan Islamabad 243700667
Bangladesh Dhaka 169384897
Afghanistan Kabul 40578842
Nepal Kathmandu 29715436
Sri Lanka Colombo 22181000
Bhutan Thimphu 780914
Maldives Male 524106

The South Asia region contains eight countries. Sorting by population descending puts India and Pakistan at the top of the list, where you would expect them.

To exclude a value instead of matching it, use <> (or the equivalent !=):

SELECT country, region
FROM countries
WHERE region <> 'North America';

3.4.2 Comparing Numeric Values

The standard comparison operators work on numeric columns: <, >, <=, >=, =, and <>.

SELECT country, region, population
FROM countries
WHERE population > 100000000
ORDER BY population DESC;

Eleven countries have populations exceeding 100 million. Notice that the result includes countries from several different regions, which you can verify by scanning the region column.

3.4.3 NULL Values and IS NULL

NULL is a special marker in SQL that represents the absence of a value. It is not the same as zero, an empty string, or a blank space. NULL means that no value was recorded for that column in that row at all.

Seven countries in the countries table have no recorded land area. You can find them using IS NULL:

SELECT country, iso3c, region, area
FROM countries
WHERE area IS NULL;
7 records
country iso3c region area
Channel Islands CHI Europe & Central Asia NA
Gibraltar GIB Europe & Central Asia NA
Hong Kong SAR, China HKG East Asia & Pacific NA
Israel ISR Middle East & North Africa NA
Kosovo XKX Europe & Central Asia NA
Macao SAR, China MAC East Asia & Pacific NA
West Bank and Gaza PSE Middle East & North Africa NA

To find rows where a value is present, use IS NOT NULL. You saw this earlier in the arithmetic section, where WHERE area IS NOT NULL excluded those seven rows before computing population density. As a standalone example:

SELECT country, iso3c, region, area
FROM countries
WHERE area IS NOT NULL
ORDER BY area DESC;
Displaying records 1 - 10
country iso3c region area
Russian Federation RUS Europe & Central Asia 16376870
China CHN East Asia & Pacific 9388210
United States USA North America 9147420

You cannot test for NULL using =. Writing WHERE area = NULL always returns zero rows, because NULL is not considered equal to anything, not even another NULL. Always use IS NULL or IS NOT NULL when checking for missing values.

3.4.4 Combining Conditions with AND and OR

A WHERE clause can include multiple conditions joined by AND or OR.

  • AND requires both conditions to be true for a row to be included.
  • OR requires at least one condition to be true.
SELECT country, population, area
FROM countries
WHERE region = 'Latin America & Caribbean'
  AND population > 10000000
ORDER BY population DESC;
Displaying records 1 - 10
country population area
Brazil 210306415 8358140
Mexico 128613117 1943950
Colombia 51737944 1109500
Argentina 45407904 2736690
Peru 33475438 1280000
Venezuela, RB 28213017 882050
Chile 19553036 743532
Guatemala 17847877 107160
Ecuador 17823897 248360
Bolivia 12077154 1083300

This query returns the 14 countries in Latin America and the Caribbean with populations above 10 million. Brazil leads comfortably, with roughly 1.6 times Mexico’s population.

When combining AND and OR in the same WHERE clause, use parentheses to make your intentions explicit. SQL evaluates AND before OR, which can produce surprising results if conditions are not grouped clearly.

-- Returns rows where region matches AND either population condition is true
WHERE region = 'South Asia'
  AND (population > 50000000 OR population < 1000000)

3.4.5 Matching Patterns with LIKE

The LIKE operator filters text columns using simple patterns. Two wildcard characters are available:

  • % matches any sequence of zero or more characters.
  • _ matches exactly one character.

The following example finds all countries whose names begin with C:

SELECT country, iso3c, capital, region
FROM countries
WHERE country LIKE 'C%'
ORDER BY country;
Displaying records 1 - 10
country iso3c capital region
Cabo Verde CPV Praia Sub-Saharan Africa
Cambodia KHM Phnom Penh East Asia & Pacific
Cameroon CMR Yaounde Sub-Saharan Africa
Canada CAN Ottawa North America
Cayman Islands CYM George Town Latin America & Caribbean
Central African Republic CAF Bangui Sub-Saharan Africa
Chad TCD N’Djamena Sub-Saharan Africa
Channel Islands CHI NA Europe & Central Asia
Chile CHL Santiago Latin America & Caribbean
China CHN Beijing East Asia & Pacific

Lowercase 'c%' would return no results here because text matching with LIKE is case-sensitive, and no country names begin with a lower-case C. If you need to perform case-insensitive pattern matching, use ILIKE instead of LIKE.

With multiple wildcards, you can search for text that occurs anywhere in a given column. In this example, by adding the % wildcard both before and after our search term, we can find any and all countries whose names contain the string “Virgin,” regardless of whether it comes at the start of the name, the end, or somewhere in the middle.

SELECT country, iso3c, capital, region
FROM countries
WHERE country LIKE '%Virgin%'
ORDER BY country;
2 records
country iso3c capital region
British Virgin Islands VGB Road Town Latin America & Caribbean
Virgin Islands (U.S.) VIR Charlotte Amalie Latin America & Caribbean

When you want to match any one of several specific values, IN is more readable than a chain of OR conditions.

3.4.7 Negating Conditions with NOT

The NOT keyword reverses the result of a condition. It can be placed in front of LIKE, IN, BETWEEN, and similar operators to express “everything except what this pattern matches.”

NOT LIKE excludes rows that match a pattern. The query below returns countries whose names do not contain the word “Republic”:

SELECT country, region
FROM countries
WHERE country NOT LIKE '%Republic%'
ORDER BY country;
Displaying records 1 - 10
country region
Afghanistan South Asia
Albania Europe & Central Asia
Algeria Middle East & North Africa
American Samoa East Asia & Pacific
Andorra Europe & Central Asia
Angola Sub-Saharan Africa
Antigua and Barbuda Latin America & Caribbean
Argentina Latin America & Caribbean
Armenia Europe & Central Asia
Aruba Latin America & Caribbean

NOT IN excludes rows whose column value matches any entry in the list:

SELECT country, region, population
FROM countries
WHERE region NOT IN ('North America', 'Europe & Central Asia')
ORDER BY population DESC;
Displaying records 1 - 10
country region population
India South Asia 1425423212
China East Asia & Pacific 1412175000
Indonesia East Asia & Pacific 278830529
Pakistan South Asia 243700667
Nigeria Sub-Saharan Africa 223150896
Brazil Latin America & Caribbean 210306415
Bangladesh South Asia 169384897
Mexico Latin America & Caribbean 128613117
Ethiopia Sub-Saharan Africa 125384287
Japan East Asia & Pacific 125124989

IS NOT NULL, introduced in the previous section, follows the same pattern and is technically a NOT form as well.

NOT can also be written as a prefix before a parenthesized condition:

WHERE NOT (region = 'North America')

This is equivalent to WHERE region <> 'North America'. The prefix form is more useful when negating a compound condition that would be awkward to rewrite with <> or other operators.

3.4.7.1 NOT IN and NULL Values

NOT IN behaves unexpectedly when NULL values are involved, and understanding this is important for using it safely.

When a NULL appears anywhere in the list passed to NOT IN, the condition returns no rows at all. To see this, try adding a NULL to an otherwise straightforward filter:

SELECT country, region, iso3c, area
FROM countries
WHERE region NOT IN ('North America', 'Europe & Central Asia', NULL);
0 records
country region iso3c area

This returns zero rows despite the fact most of the 215 countries belong to neither of the two named regions here. The reason comes down to how SQL evaluates NULL comparisons. The expression region NOT IN ('North America', 'Europe & Central Asia', NULL) expands internally to:

region <> 'North America'
AND region <> 'Europe & Central Asia'
AND region <> NULL

The third condition, region <> NULL, never evaluates to TRUE or FALSE. As covered earlier, any comparison with NULL produces NULL, meaning the result is unknown. When one part of an AND chain is unknown, the entire expression is unknown, and the row is excluded from the output. Because this applies to every row in the table, the query returns nothing.

A single NULL anywhere in the list passed to NOT IN silently causes the entire condition to return no rows. Because no error is raised, this problem can go unnoticed for a long time. When building a list for NOT IN, verify that it contains no NULL values. A later chapter covers how this same issue arises when the list comes from a subquery rather than a literal set of values, and introduces NOT EXISTS as the preferred alternative in those situations. Unlike NOT IN, NOT EXISTS handles NULL values correctly and does not produce this silent failure.

3.5 Sorting Results with ORDER BY

By default, a database returns rows in no particular order. The physical order of data in storage has nothing to do with alphabetical order, population size, or any other meaningful sequence. The ORDER BY clause lets you specify the order you want.

SELECT column1, column2
FROM table_name
ORDER BY column1 ASC;

ASC (ascending) is the default and can be omitted. Use DESC (descending) to reverse the order.

SELECT country, region, population
FROM countries
ORDER BY population DESC;
Displaying records 1 - 10
country region population
India South Asia 1425423212
China East Asia & Pacific 1412175000

Mexico Latin America & Caribbean 128613117

3.6 Limiting Results with LIMIT

The LIMIT clause caps the number of rows returned by a query. It is most useful when combined with ORDER BY, because the two together let you answer “top N” questions: the ten most populous countries, the five smallest territories by area, and so on.

SELECT country, region, population
FROM countries
ORDER BY population DESC
LIMIT 10;
Displaying records 1 - 10
country region population
India South Asia 1425423212
China East Asia & Pacific 1412175000
United States North America 333271411
Indonesia East Asia & Pacific 278830529
Pakistan South Asia 243700667
Nigeria Sub-Saharan Africa 223150896
Brazil Latin America & Caribbean 210306415
Bangladesh South Asia 169384897
Russian Federation Europe & Central Asia 144236933
Mexico Latin America & Caribbean 128613117

Using LIMIT without ORDER BY returns an arbitrary subset of rows. Which rows you get depends on how the database happens to store and retrieve data at that moment, which can vary between runs. If you need a specific subset (such as the largest, smallest, newest, or oldest), always pair LIMIT with an ORDER BY that defines what “first” means.

3.7 Putting It All Together

The clauses covered in this chapter can be combined in a single query. PostgreSQL evaluates them in a fixed logical order, regardless of how you arrange them in the text:

  1. FROM (identify the table)
  2. WHERE (filter rows)
  3. SELECT (compute and choose columns)
  4. ORDER BY (sort the result)
  5. LIMIT (cut the result to the requested number of rows)

The query below uses all of these clauses together to find the ten most populous countries in Europe and Central Asia, showing a computed population density alongside each country’s name and capital.

SELECT
    country || ' (' || iso3c || ')' AS display_name,
    capital,
    population,
    ROUND(population::numeric / area, 1) AS pop_per_sq_km
FROM countries
WHERE region = 'Europe & Central Asia'
  AND area IS NOT NULL
ORDER BY population DESC
LIMIT 10;
Displaying records 1 - 10
display_name capital population pop_per_sq_km
Russian Federation (RUS) Moscow 144236933 8.8
Turkiye (TUR) Ankara 84979913 110.4
Germany (DEU) Berlin 83797985 239.9
France (FRA) Paris 68065015 124.3
United Kingdom (GBR) London 67791000 280.2
Italy (ITA) Rome 59013667 199.6
Spain (ESP) Madrid 47759127 95.6
Ukraine (UKR) Kiev 41048766 70.8
Poland (POL) Warsaw 36821749 120.3
Uzbekistan (UZB) Tashkent 34938955 79.3

Reading a query like this from top to bottom is natural once you understand what each clause contributes. As you move through subsequent chapters and your queries grow more complex, this structured way of thinking about each clause separately will continue to serve as a reliable foundation.

3.8 SQL as a Declarative Language

Now that you hopefully understand how a basic SELECT statement is written, it’s important to understand something fundamental about how SQL thinks, because it is different from how R and Python think.

If you have experience with either of those languages, you are accustomed to writing programs that specify a sequence of steps. You tell the computer exactly what to do and in what order: load this data, filter these rows, select these columns, sort the result. This style is called imperative programming, because you are issuing commands.

SQL is declarative. Rather than specifying steps, you describe the result you want and let the database engine figure out how to produce it.

To make the contrast concrete, consider retrieving the South Asian countries sorted by population. In Python with pandas, you might write:

south_asia = df[df["region"] == "South Asia"]
result = (
    south_asia[["country", "population"]]
    .sort_values("population", ascending=False)
)

In R with dplyr:

result <- countries |>
  filter(region == "South Asia") |>
  select(country, population) |>
  arrange(desc(population))

Both of these describe a sequence of operations. The order of the instructions matches the intended order of execution.

The equivalent SQL looks like this:

SELECT country, population
FROM countries
WHERE region = 'South Asia'
ORDER BY population DESC;

The SQL query does not describe steps. It describes a result: “I want the country and population columns, from the countries table, restricted to rows where region is 'South Asia', sorted by population descending.” The database engine reads that description and decides for itself how to retrieve the data efficiently. It may use an index, reorder internal operations, or parallelize work across processors. You do not control those decisions, and most of the time you do not need to.

3.8.1 Set-Based Thinking

One practical consequence of SQL’s declarative nature is that it operates on sets of rows rather than individual records. When you write a WHERE clause, you are not writing a loop that examines each row in turn. You are describing a condition that applies to the entire table at once, and the database evaluates it however it finds most efficient.

This can feel unfamiliar at first. In Python or R, reaching for a loop or an apply-style function is instinctive. In SQL, you resist that instinct and instead describe what the complete result set should look like. Queries that try to mimic row-by-row processing are almost always slower and harder to read than queries that embrace the set-based model.

3.8.2 The Logical Order of Execution

There is one aspect of SQL’s declarative model that surprises many people coming from R or Python: the order in which you write a query’s clauses is not the order in which the database evaluates them.

A query is written as:

SELECT ...
FROM ...
WHERE ...
ORDER BY ...
LIMIT ...

But the database processes the clauses in this logical order:

  1. FROM (identify the source table)
  2. WHERE (filter rows)
  3. SELECT (compute and choose columns)
  4. ORDER BY (sort the result)
  5. LIMIT (trim to the requested number of rows)

The most common practical consequence of this is that column aliases defined in SELECT cannot be referenced in a WHERE clause, because WHERE runs before SELECT. The following query fails for exactly that reason:

-- This fails: 'density' is not defined when WHERE runs
SELECT population / area AS density
FROM countries
WHERE density > 100;

You have to repeat the expression in WHERE instead:

SELECT population / area AS density
FROM countries
WHERE population / area > 100;

ORDER BY is an exception: most databases, including PostgreSQL, allow ORDER BY to reference aliases defined in SELECT, because ORDER BY is evaluated after SELECT. That is why ORDER BY density would work in the query above even though WHERE density does not.

Keeping this logical order in mind will make otherwise cryptic error messages much easier to diagnose as your queries grow more complex.
