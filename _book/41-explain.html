22  Reading Query Plans

22.1 Learning Objectives

By the end of this chapter, students will be able to:

  • Use EXPLAIN to display the execution plan for a SQL statement without running it.
  • Use EXPLAIN ANALYZE to execute a query and compare estimated versus actual row counts and timing.
  • Interpret the cost notation (cost=startup..total rows=N width=W) in a query plan node.
  • Identify common plan node types including Sequential Scan, Index Scan, Bitmap Heap Scan, Hash Join, and Nested Loop.
  • Explain when the query planner chooses a sequential scan over an index scan and why that choice may be correct.
  • Diagnose a slow query by locating the highest-cost node or the largest discrepancy between estimated and actual rows in a plan.
  • Use EXPLAIN (ANALYZE, BUFFERS) to inspect cache hit rates and buffer usage alongside timing information.

SQL queries do not have a single fixed way of executing. When you submit a SELECT statement, PostgreSQL does not simply start reading rows from top to bottom. It generates a query plan: a tree of operations that describes exactly how it will retrieve, filter, join, sort, and aggregate the data. Understanding how to read that plan is what separates someone who writes SQL from someone who understands how their queries actually run.

This chapter uses the nycflights database. The flights table contains 104,662 departure records from LaGuardia Airport, joined by foreign keys to small lookup tables for airlines, airports, and aircraft. That combination of a large central table and small reference tables produces a variety of interesting plan shapes.

22.2 The EXPLAIN Command

EXPLAIN preceded a SELECT statement returns the query plan without executing the query. No rows are read from disk, no results are returned — only the plan.

EXPLAIN SELECT * FROM flights WHERE carrier = 'UA';
2 records
QUERY PLAN
Seq Scan on flights (cost=0.00..3178.28 rows=8132 width=81)
Filter: ((carrier)::text = ‘UA’::text)

The output is a tree of plan nodes, each representing one operation. This plan has two nodes. The outer node is Seq Scan on flights, which reads the entire table row by row. Indented below it is a Filter, which is not a separate node but a condition applied inside the scan.

22.3 Anatomy of a Plan Node

Every node in a query plan carries a cost annotation in parentheses:

Seq Scan on flights  (cost=0.00..3178.28 rows=8132 width=81)

Each field has a specific meaning.

cost=startup..total. The cost is a dimensionless unit that represents estimated work. The startup cost is how much work is needed before the first row can be returned. The total cost is the estimated work to return all rows. These numbers are not milliseconds — they are relative units the planner uses to compare strategies against each other. A plan with total cost 100 is estimated to be roughly ten times faster than one with total cost 1000.

rows. The planner’s estimate of how many rows this node will produce. PostgreSQL derives these estimates from table statistics collected by ANALYZE — a point returned to below.

width. The average width in bytes of each output row. For SELECT * on flights, each row is about 81 bytes.

22.4 Sequential Scan

A sequential scan reads every page of the table from start to finish, applying any filter condition to each row as it goes. For the query above, PostgreSQL chose a sequential scan even though the filter carrier = 'UA' discards most rows. That is not a mistake.

EXPLAIN ANALYZE executes the query and augments each node with actual measurements:

EXPLAIN ANALYZE SELECT * FROM flights WHERE carrier = 'UA';
5 records
QUERY PLAN
Seq Scan on flights (cost=0.00..3178.28 rows=8132 width=81) (actual time=0.037..19.601 rows=8044 loops=1)
Filter: ((carrier)::text = ‘UA’::text)
Rows Removed by Filter: 96618
Planning Time: 0.087 ms
Execution Time: 19.974 ms

The actual time=0.088..31.365 rows=8044 loops=1 annotation shows that the scan took about 31 milliseconds and returned 8,044 rows. The Rows Removed by Filter: 96618 line shows how many rows were discarded.

United Airlines operates about 7.7 percent of the flights in this dataset. Fetching 7.7 percent of a table by scanning it sequentially is often cheaper than using an index, because index lookups involve random I/O: jumping to index pages, then jumping to heap pages, potentially many times. For large fractions of a table, the sequential scan’s predictable, linear I/O pattern is faster. The planner made the right call here.

The estimated row count (8,132) and the actual count (8,044) are close but not identical. PostgreSQL’s estimates are based on column statistics sampled periodically by ANALYZE, not on exact counts. Small discrepancies are normal and harmless. Large discrepancies are a signal to investigate — more on this shortly.

22.5 Index Scan

When a query is highly selective — matching only a handful of rows out of millions — the planner switches to an index scan. The flights table has a composite primary key on (year, month, day, carrier, flight, origin). A lookup that supplies all six components narrows the search to a single row.

EXPLAIN ANALYZE
SELECT *
FROM   flights
WHERE  year = 2013 AND month = 6 AND day = 15
   AND carrier = 'AA' AND flight = 301 AND origin = 'LGA';
4 records
QUERY PLAN
Index Scan using flights_pkey on flights (cost=0.42..8.45 rows=1 width=81) (actual time=0.044..0.045 rows=1 loops=1)
Index Cond: ((year = 2013) AND (month = 6) AND (day = 15) AND ((carrier)::text = ‘AA’::text) AND (flight = 301) AND ((origin)::text = ‘LGA’::text))
Planning Time: 0.187 ms
Execution Time: 0.092 ms

The plan is Index Scan using flights_pkey. The total cost is around 8 — compared to 3,178 for the sequential scan above. PostgreSQL traverses the B-tree index to find the matching leaf entry and then fetches the single heap page that contains that row. Planning and execution together take under 2 milliseconds.

The key difference from a sequential scan is that the index gives the planner a direct address for the row rather than requiring it to examine the full table.

22.6 Row Count Estimates and Statistics

The planner builds its estimates from statistics stored in pg_statistic, which is populated by the ANALYZE command (run automatically in the background by autovacuum, and manually with ANALYZE table_name). When those statistics are stale or missing, estimates can diverge dramatically from reality.

Consider joining flights to airlines to count flights per carrier:

EXPLAIN ANALYZE
SELECT f.carrier, a.name, COUNT(*)
FROM   flights f
JOIN   airlines a ON f.carrier = a.carrier
GROUP  BY f.carrier, a.name
ORDER  BY count(*) DESC;
14 records
QUERY PLAN
Sort (cost=4060.54..4061.06 rows=208 width=31) (actual time=53.389..53.395 rows=13 loops=1)
Sort Key: (count(*)) DESC
Sort Method: quicksort Memory: 25kB
-> HashAggregate (cost=4050.45..4052.53 rows=208 width=31) (actual time=53.329..53.337 rows=13 loops=1)
Group Key: f.carrier, a.name
Batches: 1 Memory Usage: 40kB
-> Hash Join (cost=1.36..3265.49 rows=104662 width=23) (actual time=0.051..30.807 rows=104662 loops=1)
Hash Cond: ((f.carrier)::text = (a.carrier)::text)
-> Seq Scan on flights f (cost=0.00..2916.62 rows=104662 width=3) (actual time=0.008..7.335 rows=104662 loops=1)
-> Hash (cost=1.16..1.16 rows=16 width=23) (actual time=0.019..0.020 rows=16 loops=1)
Buckets: 1024 Batches: 1 Memory Usage: 9kB
-> Seq Scan on airlines a (cost=0.00..1.16 rows=16 width=23) (actual time=0.005..0.007 rows=16 loops=1)
Planning Time: 1.766 ms
Execution Time: 53.629 ms

Scan the plan carefully. The Seq Scan on airlines node reads the table fine — 16 rows returned — but the estimate says rows=310. PostgreSQL thought airlines had about 310 rows. The table actually has 16.

That mismatch propagates upward: the HashAggregate node estimated 2,600 output groups but produced only 13. The planner’s statistics for the airlines table are stale. Running ANALYZE corrects them:

EXPLAIN
SELECT f.carrier, a.name, COUNT(*)
FROM   flights f
JOIN   airlines a ON f.carrier = a.carrier
GROUP  BY f.carrier, a.name
ORDER  BY count(*) DESC;
9 records
QUERY PLAN
Sort (cost=4060.54..4061.06 rows=208 width=31)
Sort Key: (count(*)) DESC
-> HashAggregate (cost=4050.45..4052.53 rows=208 width=31)
Group Key: f.carrier, a.name
-> Hash Join (cost=1.36..3265.49 rows=104662 width=23)
Hash Cond: ((f.carrier)::text = (a.carrier)::text)
-> Seq Scan on flights f (cost=0.00..2916.62 rows=104662 width=3)
-> Hash (cost=1.16..1.16 rows=16 width=23)
-> Seq Scan on airlines a (cost=0.00..1.16 rows=16 width=23)

The estimate for airlines now matches reality. When a plan behaves unexpectedly or a query is slower than it should be, stale statistics are one of the first things to check.

22.7 Join Strategies

PostgreSQL implements three join algorithms. The planner chooses among them based on the estimated sizes of the inputs and the available indexes.

22.7.1 Hash Join

A hash join works in two phases. First, PostgreSQL builds a hash table from the smaller of the two inputs. Then it scans the larger input and probes the hash table for matching rows. Hash joins are efficient when one side fits comfortably in memory and the join condition uses equality.

The flights JOIN airlines plan above uses a hash join. The airlines table (16 rows) is hashed in memory, and the full flights table is scanned once to probe it. Building the hash table costs almost nothing; the scan of flights dominates the total cost.

22.7.2 Nested Loop

A nested loop iterates over the outer input and, for each row, looks up matching rows in the inner input — ideally using an index. Nested loops are efficient when the outer input is small and the inner lookup is fast.

A query that filters flights heavily before joining to airlines produces a nested loop:

EXPLAIN
SELECT f.*, a.name
FROM   flights f
JOIN   airlines a ON f.carrier = a.carrier
WHERE  f.month = 7 AND f.dep_delay > 120;
8 records
QUERY PLAN
Nested Loop (cost=0.15..3448.21 rows=226 width=101)
-> Seq Scan on flights f (cost=0.00..3439.93 rows=226 width=81)
Filter: ((dep_delay > 120) AND (month = 7))
-> Memoize (cost=0.15..0.22 rows=1 width=23)
Cache Key: f.carrier
Cache Mode: logical
-> Index Scan using airlines_pkey on airlines a (cost=0.14..0.21 rows=1 width=23)
Index Cond: ((carrier)::text = (f.carrier)::text)

With only around 226 matching flights after the filter, iterating over them and looking up each carrier in the airlines index is faster than building a full hash table. Notice the Memoize node wrapping the index scan: PostgreSQL caches the result of each carrier lookup so it does not repeat the same index probe for the same carrier twice.

22.7.3 Merge Join

A merge join requires both inputs to be sorted on the join key. PostgreSQL advances through both sorted streams simultaneously, pairing matching rows. Merge joins are not demonstrated here because the nycflights schema does not produce one naturally without additional indexes, but you will see them in schemas where large, pre-sorted tables are joined on a key that both sides have indexed.

22.8 Bitmap Heap Scan

Between the sequential scan (reads everything) and the index scan (fetches one or a few rows directly) lies a third strategy: the bitmap scan.

Without any index on dep_delay, a filter like dep_delay > 120 forces a sequential scan:

EXPLAIN ANALYZE SELECT * FROM flights WHERE dep_delay > 120;
5 records
QUERY PLAN
Seq Scan on flights (cost=0.00..3178.28 rows=2778 width=81) (actual time=0.019..5.991 rows=2791 loops=1)
Filter: (dep_delay > 120)
Rows Removed by Filter: 101871
Planning Time: 0.074 ms
Execution Time: 6.101 ms

About 2,791 rows match the condition — roughly 2.7 percent of the table. The planner scans all 104,662 rows to find them, taking around 11 milliseconds.

Adding an index on dep_delay changes the plan:

EXPLAIN ANALYZE SELECT * FROM flights WHERE dep_delay > 120;
7 records
QUERY PLAN
Bitmap Heap Scan on flights (cost=33.82..2029.37 rows=2778 width=81) (actual time=0.572..1.648 rows=2791 loops=1)
Recheck Cond: (dep_delay > 120)
Heap Blocks: exact=890
-> Bitmap Index Scan on idx_flights_dep_delay (cost=0.00..33.13 rows=2778 width=0) (actual time=0.431..0.432 rows=2791 loops=1)
Index Cond: (dep_delay > 120)
Planning Time: 0.412 ms
Execution Time: 1.895 ms

The plan is now a Bitmap Heap Scan. It works in two phases. First, the Bitmap Index Scan traverses the index and builds an in-memory bitmap marking which heap pages contain matching rows. Then the Bitmap Heap Scan fetches only those pages, rechecking the filter condition for each row it finds. The execution time drops from about 11 milliseconds to around 2.

For a highly selective filter — dep_delay > 300, which matches only 228 rows — the same index is even more effective:

EXPLAIN ANALYZE SELECT * FROM flights WHERE dep_delay > 300;
7 records
QUERY PLAN
Bitmap Heap Scan on flights (cost=5.90..607.22 rows=208 width=81) (actual time=0.060..0.244 rows=228 loops=1)
Recheck Cond: (dep_delay > 300)
Heap Blocks: exact=148
-> Bitmap Index Scan on idx_flights_dep_delay (cost=0.00..5.85 rows=208 width=0) (actual time=0.037..0.038 rows=228 loops=1)
Index Cond: (dep_delay > 300)
Planning Time: 0.083 ms
Execution Time: 0.350 ms

Only 148 heap pages need to be fetched, and the query runs in well under a millisecond.

The planner chooses between these three scan strategies automatically based on estimated selectivity:

Selectivity Strategy
Large fraction of the table Sequential Scan
Moderate fraction, range condition Bitmap Heap Scan
Very few rows, equality or narrow range Index Scan

These thresholds are not fixed — they depend on table size, index size, and the cost parameters configured for the server.

22.9 EXPLAIN (ANALYZE, BUFFERS)

The BUFFERS option adds cache statistics to each node, showing how many 8 KB buffer pages were found in shared memory (shared hit) versus read from disk (shared read).

EXPLAIN (ANALYZE, BUFFERS)
SELECT f.dest, a2.name AS dest_name,
       COUNT(*)                             AS n_flights,
       ROUND(AVG(f.arr_delay)::numeric, 1)  AS avg_arr_delay
FROM   flights f
JOIN   airports a2 ON f.dest = a2.faa
WHERE  f.dep_delay > 30
GROUP  BY f.dest, a2.name
ORDER  BY avg_arr_delay DESC
LIMIT  10;
Displaying records 1 - 25
QUERY PLAN
Limit (cost=3884.90..3884.93 rows=10 width=64) (actual time=11.831..11.835 rows=10 loops=1)
Buffers: shared hit=1892
-> Sort (cost=3884.90..3918.15 rows=13298 width=64) (actual time=11.830..11.832 rows=10 loops=1)
Sort Key: (round(avg(f.arr_delay), 1)) DESC
Sort Method: top-N heapsort Memory: 26kB
Buffers: shared hit=1892
-> HashAggregate (cost=3398.07..3597.54 rows=13298 width=64) (actual time=11.715..11.771 rows=64 loops=1)
Group Key: f.dest, a2.name
Batches: 1 Memory Usage: 417kB
Buffers: shared hit=1889
-> Hash Join (cost=51.80..3265.09 rows=13298 width=28) (actual time=0.428..9.004 rows=13136 loops=1)
Hash Cond: ((f.dest)::text = (a2.faa)::text)
Buffers: shared hit=1889
-> Seq Scan on flights f (cost=0.00..3178.28 rows=13298 width=8) (actual time=0.018..6.018 rows=13136 loops=1)
Filter: (dep_delay > 30)
Rows Removed by Filter: 91526
Buffers: shared hit=1870
-> Hash (cost=33.58..33.58 rows=1458 width=24) (actual time=0.393..0.394 rows=1458 loops=1)
Buckets: 2048 Batches: 1 Memory Usage: 97kB
Buffers: shared hit=19
-> Seq Scan on airports a2 (cost=0.00..33.58 rows=1458 width=24) (actual time=0.003..0.132 rows=1458 loops=1)
Buffers: shared hit=19
Planning:
Buffers: shared hit=82
Planning Time: 1.057 ms

The flights scan reports shared hit=1870 and no reads, meaning the entire table was already in PostgreSQL’s shared buffer cache from prior queries in this chapter. The airports scan reports shared read=19, meaning it needed to load 19 pages from disk — the airports table had not been accessed recently enough to remain cached.

Buffer statistics are most useful when diagnosing queries that are slow because they repeatedly read from disk. A query showing many shared read entries on a table that should be cached is a signal that the database’s shared_buffers setting may be undersized, or that the table is simply too large to fit in memory.

22.10 Finding the Bottleneck

When a query is slower than expected, reading its EXPLAIN ANALYZE output from the bottom up reveals where the time is actually being spent.

Start at the leaves. The leaf nodes are where data enters the plan — scans and index lookups. Their actual row counts and times tell you whether the planner’s estimates were reasonable.

Look for large estimate discrepancies. A node where estimated rows and actual rows differ by a factor of ten or more means the planner made a decision under false assumptions. The plan may be sub-optimal as a result. Running ANALYZE on the affected table is the first corrective step.

Find the most expensive node. Each node’s actual time is cumulative: it includes all the work done by its children. The node with the highest actual time is where optimization effort will have the most impact. If that node is a sequential scan and the filter is highly selective, a new index is worth considering — which is the subject of the next chapter.

Check for loops. The loops=N annotation on a node shows how many times that node executed. A nested loop with a large outer input will execute the inner node many times. If the inner node is doing expensive work on each iteration, that multiplication effect dominates the total time.
